# アイキャッチ画像生成ワークフロー

## 実行入口

AIで記事のアイキャッチを作成・再開・採用するときは、リポジトリスキル `$jekyll-eyecatch-backfill` を使用する。

```text
$jekyll-eyecatch-backfill
```

詳細な対象選定、状態判定、候補生成、上限到達時の再開、日英ペア同期、採用処理、生成HTML確認はスキルを正本とする。この文書は、スキルが守る方針と人が行う判断だけを定める。

## 方針

- 1回につき日本語記事1本、または対応する日英ペア1組だけを扱う。
- 記事のカテゴリではなく、記事固有の出来事または一つの強い比喩を描く。
- 同じ記事について媒体・配色・構図・抽象度が異なる候補を最低4つ作る。seed違いだけでは4案と数えない。
- リソグラフ／スクリーンプリントを基準案として残し、標準スタイルから2案、記事固有の実験枠を1案以上含める。
- 画像内へ記事タイトル、文字、数字、ロゴ、透かし、偽UI、コードを入れない。
- 暗いネオン背景、発光するネットワーク、AIロボットなどの汎用的な技術表現へ置き換えない。
- 16:9、1200×675px、縮小時に主役が残る構図を標準とする。小さい画像を拡大しない。

## 候補と採用の境界

選定前の生成物は `.eyecatch-work/YYYY-MM-DD-slug/` に置き、フロントマターと公開用 `assets/` を変更しない。既存ファイルを上書きせず、再生成時は `-v2` などのサフィックスを付ける。

エージェントは候補画像を開いたり、スクリーンショットや比較シートを作ったり、視認評価したりしない。ファイルの存在、形式、寸法、サイズだけを確認し、画像の判断はユーザーへ委ねる。

ユーザーは次を判断する。

1. どの候補または方向性を残すか
2. 記事固有の出来事を最も正しく表しているか
3. 320×180程度でも主役が分かるか
4. そのまま採用するか、一要素だけ変えて反復するか

採用または明示的な許可より前に、主役が切れる可能性のあるクロップを行わない。

## 停止条件

- HTTP 429、quota、rate-limit、capacityエラーが出た時点で停止する。
- 生成済み候補と未生成候補を報告する。
- 別モデルや別サービスへ無断で切り替えない。
- リセット時刻を取得できる場合はJSTで報告する。
- 生成済み候補があれば、回復を待つ前に方向性のフィードバックを求める。

## 採用後の完了条件

- 採用画像だけを `assets/images/posts/YYYY-MM-DD-slug/` に置く。
- フロントマターには `image.path` と内容を説明する `image.alt` を設定し、`banner` は使わない。
- 日英ペアでは同じ画像を共有し、altは各言語で記述する。
- 不採用候補は `.eyecatch-work/` に残し、コミット対象にしない。
- Jekyllをビルドし、記事ヘッダー、Home、OGP、Twitter Cardが採用画像とaltを参照することを生成HTMLで確認する。
- エージェントはレンダリング結果を視認評価しない。表示崩れや縮小時の読みやすさはユーザーが確認する。

PNGが2MBを超える場合はサイズを報告する。ロスレス再圧縮やWebP化はユーザーの判断後に行う。
