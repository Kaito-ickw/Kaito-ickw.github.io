# アイキャッチ画像生成ワークフロー

## 目的

記事固有の内容が小さなサムネイルでも伝わるアイキャッチを作る。画像生成モデルが出しやすい「暗い背景、ネオン、発光する回路、抽象的なネットワーク」に寄せること自体を目的にしない。

完成品を1枚だけ生成して採用しない。同じ記事について視覚表現の異なる候補を最低4つ作り、人が方向性を選んでから仕上げる。

## なぜ汎用的な技術画像になるのか

次の条件が重なると、内容に関係なく似た画像になりやすい。

- 「AI」「技術」「安全」「接続」のような抽象語だけで主題を指定する
- 最初から全記事へ同じ配色と画風を適用する
- `dark navy`、`neon cyan`、`glowing network`のような生成モデルの定番表現を主要な指示にする
- 記事内で何が起きるのかではなく、記事のカテゴリを描かせる
- 1記事につき1枚だけ生成し、比較せず採用する

「2つのAIエージェントが接続する」ではなく、「停止した処理へメッセージだけが届き、別の起動トリガーが必要になる」のように、記事固有の出来事を描く必要がある。

## 基本原則

### カテゴリではなく出来事を描く

生成前に、記事の内容を次の形へ変換する。

```text
登場物: 画面内に存在する具体的な物
行動: 何が起きているか
障害: 期待どおり進まない原因
変化: 障害が見える、または解消される瞬間
```

例:

```text
登場物: 2つの作業機械、完了した荷物、起動スイッチ
行動: 一方の機械から荷物が送られる
障害: 受信側は停止しており、荷物だけが扉の前に残る
変化: 別のスイッチが受信側を起動し、双方向の受け渡しが始まる
```

画像をタイトルなしで一文説明できなければ、生成前の視覚設計がまだ抽象的である。

### 生成モデルに代替させない物を明記する

記事固有の場面に加えて、安易な代替表現を禁止する。

```text
Do not substitute the story with a generic glowing network,
an abstract technology ring, a robot portrait, or a blue circuit background.
```

禁止事項だけでは画像は決まらない。必ず先に具体的な場面を指定し、その後へ禁止事項を書く。

### 画風ではなく4軸を変える

候補間では、少なくとも次の4軸を変える。

| 軸 | 例 |
| :--- | :--- |
| 媒体 | リソグラフ、紙コラージュ、ミニチュア、フラットイラスト |
| 配色 | 2色刷り、暖色、工業色、強い補色 |
| 構図 | 斜めの動線、左右対比、接写、単一の大きな比喩 |
| 抽象度 | 具体的な場面、物による比喩、記号化したポスター |

同じプロンプトのseed違いを4案とは数えない。

## 標準フロー

### 1. 記事ごとに処理する

各記事について最低4案を生成する。ただし、複数記事の候補を一つの大きなバッチで生成せず、1記事分の生成、ファイル存在確認、ユーザーへの選定依頼を終えてから次の記事へ進む。

これにより、途中で生成枠へ到達しても記事単位で候補が揃い、ユーザーが選定できない半端な状態を減らせる。

### 2. ビジュアルブリーフを作る

本文を読み、生成前に以下を記述する。

```text
記事の約束: 読者がこの記事から理解できること
一枚で描く出来事: 登場物、行動、障害、変化
サムネイルで残す主役: 1〜2個
使わない代替表現: 記事を汎用化してしまう表現
文字なしで伝える範囲: 画像だけで理解させる内容
```

図解のように複数の論点を1枚へ詰め込まない。アイキャッチでは、一つの出来事または一つの強い比喩に絞る。

### 3. 最低4スタイルを別々に生成する

スタイル探索時の標準候補は次の4つとする。

1. リソグラフ／スクリーンプリント
   - このブログでの第一候補
   - 2〜3色、インクの粒子、強い形、雑誌的な編集感
2. 紙コラージュ
   - 物の関係を読み取りやすく、親しみがある
   - 紙の層、切り口、影を使う
3. ミニチュア・ジオラマ
   - 具体的な出来事と物語を最も表現しやすい
   - 素材感、照明、接写で視線を作る
4. フラットな編集ポスター
   - 一つの比喩を大きく見せる
   - 画像内にポスター用紙や額縁を描かず、16:9の画面全体を作品にする

長期的に同じ4種へ固定しない。リソグラフを基準案として残し、標準スタイルから2案、記事に合う実験枠を1案以上含める。実験例は、設計図、粘土模型、木版画、写真コラージュ、断面模型である。

### 4. 候補ファイルとして保存する

選定前は記事のフロントマターを変更しない。既存画像も上書きしない。候補と旧生成物は公開用の`assets/`ではなく、Git管理外の`.eyecatch-work/`へ置く。

```text
.eyecatch-work/YYYY-MM-DD-slug/
  candidates/
    candidate-01-risograph.png
    candidate-02-paper-collage.png
    candidate-03-miniature-diorama.png
    candidate-04-editorial-poster.png
  archive/
    eyecatch-previous.png
```

再生成時は`-v2`などのサフィックスを付ける。ユーザーが採用を決めた後にだけ、選択画像を`assets/images/posts/YYYY-MM-DD-slug/eyecatch.png`へコピーする。

`.eyecatch-work/`は`.gitignore`で除外する。過去にGit管理されていた採用画像はGit履歴から復元できるため、同じ画像をリポジトリ内へ重複保存しない。Git管理前の候補や、比較のため一時的に残したい生成物だけをローカルの`archive/`へ置く。

### 5. `agy`へ渡すプロンプトを組み立てる

`agy`では候補ごとに別の画像生成を実行する。複数案を1枚の画像内へ並べる指示はしない。

```text
Use case: stylized-concept
Asset type: Jekyll article eyecatch and small Home-page thumbnail

Article promise:
<記事から理解できること>

Core visual story:
<登場物、行動、障害、変化を具体的に記述>

Style and medium:
<この候補だけの媒体、素材、配色>

Composition:
16:9 landscape. One immediately readable story.
Keep the primary subject inside the central safe area.
The artwork itself fills the entire canvas.

Constraints:
- exactly 1200x675 PNG after clean scale and crop
- readable at approximately 320x180
- no title text, letters, numbers, logo, watermark, fake UI, or code
- strong silhouette and no unnecessary detail

Avoid:
- generic glowing network or circular connection diagram
- generic robot portrait
- generic dark navy and neon technology background
- an infographic containing several separate explanations
- a framed poster or paper sheet rendered inside the 16:9 canvas
```

コマンドの基本形は次のとおりである。

```bash
agy --print-timeout 30m -p "<候補ごとの構造化プロンプト>"
```

非対話実行のために権限自動承認が必要な環境でも、`--dangerously-skip-permissions`を常用しない。作業対象と出力先を限定し、ユーザーまたは実行環境の承認を得た場合だけ使用する。

### 6. エージェントの確認はファイル生成までに留める

画像候補の視認チェックはユーザーが行う。エージェントは、生成後の候補画像を開いて内容を評価しない。画像を表示・スクリーンショット取得・比較シート作成・サムネイル化して確認する作業は、画像理解のためのコンテキストを消費するためである。

エージェントが行う確認は次の範囲に限定する。

- 候補ファイルが生成され、`.eyecatch-work/YYYY-MM-DD-slug/candidates/`に存在する
- ファイル名が候補番号と画風を識別できる
- 必要な場合だけ、`file`や`ffprobe`などのメタデータで形式と寸法を確認する

次の判断はユーザーへ委任する。

- 文字、ロゴ、透かし、破綻した形がないか
- 主役が端で切れていないか
- 記事固有性、即読性、視覚的な引き、物語性、縮小耐性があるか
- どの案を採用または再生成するか

#### リサイズとファイル形式

候補はすべて最終利用可能な1200×675pxで保存する。選定後に同じ画像を再生成したり、小さい候補を拡大したりしない。

生成結果が16:9でない場合でも、エージェント判断でクロップしない。リサイズやクロップは主役が切れる可能性があるため、ユーザーが採用候補を選んだ後、または明示的に許可した場合だけ行う。実行する場合も画像を開いて評価せず、メタデータとユーザー指定に基づいて処理する。

```bash
ffmpeg -i input.png \
  -vf "scale=1200:675:force_original_aspect_ratio=increase,crop=1200:675" \
  -frames:v 1 output.png
```

- 入力が1200×675px未満なら拡大せず、16:9で再生成する
- 中央クロップで主役が切れるかどうかの判断はユーザーに委任する
- 採用画像の標準は当面PNGとする。記事表示とOGPが同じ画像URLを使うためである
- PNGが2MBを超える場合は、ファイルサイズだけを報告し、ロスレス再圧縮またはWebP化はユーザー判断で行う
- WebPを採用する場合はquality 90を基準とし、記事、Home、OGPの生成HTMLを確認する

WebPへの変換例:

```bash
ffmpeg -i eyecatch.png -c:v libwebp -quality 90 eyecatch.webp
```

ユーザーが視認するときの評価観点は次のとおりである。

| 評価項目 | 確認する問い |
| :--- | :--- |
| 記事固有性 | 別の記事へそのまま流用できてしまわないか |
| 即読性 | 2秒で主役と出来事を認識できるか |
| 視覚的な引き | 一覧の中で目が止まる形や色があるか |
| 物語性 | 記事の問題または変化を一文で説明できるか |
| 縮小耐性 | 細部を失っても構図が成立するか |

「きれい」「生成品質が高い」だけでは採用しない。

### 7. 人へ比較を依頼する

エージェントは候補を視認評価しない。候補ファイルのパス、生成に使った画風、必要なら非視覚メタデータだけを提示し、選定をユーザーへ依頼する。

```text
- candidate-01-risograph.png: リソグラフ／スクリーンプリント
- candidate-02-paper-collage.png: 紙コラージュ
- candidate-03-miniature-diorama.png: ミニチュア・ジオラマ
- candidate-04-editorial-poster.png: 編集ポスター
```

ユーザーには次の3点を確認する。

1. どの画風を基準にしたいか
2. どの案が記事内容を最も正しく表しているか
3. 残したい要素と変えたい要素は何か

### 8. 選ばれた方向だけ反復する

次の生成では4案すべてを作り直さない。選ばれた1〜2案について、一度に一つの要素だけ変える。

- 場面は維持して配色だけ変える
- 画風は維持して主役を大きくする
- 構図は維持して障害を読みやすくする

場面、画風、配色、構図を同時に変更すると、どの指示が改善につながったか分からなくなる。

### 9. 採用後に記事へ反映する

ユーザーの採用後に次を行う。

1. 選択画像を`eyecatch.png`または`eyecatch.webp`へ整理する
2. 内容を説明するaltテキストを書く
3. 日英ペアの記事では両方のフロントマターへ適切な言語のaltを設定する
4. 不採用候補は`.eyecatch-work/`へ残し、コミット対象から外す
5. Jekyllをビルドする
6. 生成HTMLで、記事ヘッダー、Homeのカード、OGPメタデータが採用画像のパスとaltを参照していることを確認する

ここでもエージェントはレンダリング結果を視認評価しない。表示崩れ、トリミング、サムネイルでの読みやすさはユーザー確認とする。

## 停止条件

- 画像生成サービスがHTTP 429を返した場合は、その時点で停止する
- 生成済み候補と未生成候補を分けて報告する
- 別の生成サービスや別モデルへ無断で切り替えない
- 生成枠の回復を待つ前に、生成済み候補から方向性のフィードバックを得る

## 採用判定の短いチェック

```text
[ ] ユーザーが、記事タイトルを隠しても描かれた出来事を説明できると判断した
[ ] ユーザーが、「発光する接続」「AIロボット」以外の固有要素があると判断した
[ ] ユーザーが、320×180でも主役が分かると判断した
[ ] ユーザーが、同じ記事について異なる画風を比較した
[ ] ユーザー選定前にフロントマターを変更していない
[ ] 採用後にHome、記事、OGPのHTML参照を確認した
```
