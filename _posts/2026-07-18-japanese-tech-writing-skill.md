「重要なのは〜である」「本章では〜を見ていく」「〜に他ならない」。LLM に日本語の文章を書かせたことがあれば、この口調に見覚えがあるはずだ。内容は合っているのに、どこか信用ならない。この「LLM っぽさ」の正体を条項単位で名指しし、禁止する SKILL.md が公開されて話題になっている。

ラムダノートの編集者・鹿野桂一郎さんが公開した [japanese-tech-writing](https://gist.github.com/k16shikano/fd287c3133457c4fd8f5601d34aa817d)（日本語技術文書の文章規範）だ。技術書の編集で使われてきた文章規範を、LLM が従える形式に書き下ろしたもので、「プロの編集者が編集知識を SKILL として公開した」という点で反響を呼んだ。作者自身も「SKILL は育てていくもの。手元で自由に改変して使ってほしい」と述べている。

実際に Claude Code に読み込ませ、同じテーマの文章をスキルなし・ありで書かせて比べた。

---

## 規範の中身

SKILL.md は約 9,000 字で、次の 8 章からなる。

| 章 | 規定していること |
| :--- | :--- |
| 整形 | 一文一行、ダッシュ・中黒の禁止、太字と「」の使い分け |
| 段落と論証の構成 | パラグラフライティング。段落は論証の一歩 |
| 論証の厳密さ | ツッコミどころの除去。推量を機械的に断定へ変えない |
| 読み手の負荷の管理 | 記憶と注意を有限の資源として扱う |
| 視点と語り | 行為者を主語に。曖昧語への後退の禁止 |
| 演出の抑制 | 決め台詞・太字強調・煽りの節度 |
| LLM っぽい表現の禁止 | 空虚な型の名指しと禁止 |
| 冗長の排除 | 同じ主張は一度だけ書く |

一読して分かるのは、これが「良い文章を書くコツ」の一般論ではなく、原稿レビューで実際に入る赤字を規則化したものだということだ。条項の多くが悪い例・良い例のペアで書かれている。

いくつか条項を拾ってみる。

話題の中心になったのは「LLM っぽい表現の禁止」の章だろう。「重要なのは〜である」「正面から扱う」「多角的」「掘り下げる」「さらに・また・加えての連打」などが、「論点を増やさず『ちゃんと書いている感』だけを付ける LLM 口調」として列挙されている。禁止するだけでなく、なぜ空虚なのか（何をどう見たかを書かない、姿勢だけを宣言する）が添えられているので、チェックリストとしてそのまま使える。

一方で、規範は「引き締めろ」一辺倒ではない。「論証の厳密さ」の章には、推量・可能性・読者の疑念として書かれた文を機械的に断定へ変えるな、という条項がある。「かもしれない」を削ってよいのは根拠なく主張を弱めている場合だけで、不確実な内容はその不確実性を保て、と明記されている。LLM に「簡潔にして」と頼むとニュアンスごと削られる経験をした人には、この方向の歯止めがある意味が分かるはずだ。

細かいところでは、日本語の地の文での em ダッシュ（—）と並列の中黒（・）の禁止も入っている。どちらも LLM 特有というより、和文組版で編集者が直してきた箇所で、規範全体が「編集の赤字」の集積であることがよく表れている。

## セットアップ

Gist の SKILL.md をスキルディレクトリに置くだけでよい。

```
~/.claude/skills/
└── japanese-tech-writing/
    └── SKILL.md
```

プロジェクト単位で使うならリポジトリの `.claude/skills/` に置く。frontmatter の `description` に「日本語で技術書の章、草稿、記事、解説文を書くとき、または推敲・リライトするときに使用する」とあるので、該当タスクで自動的に読み込まれる。スキルの仕組みは[Claude Code のスキルを今から使いはじめる]({% post_url 2026-06-10-claude-code-skills-guide %})に書いた。

## スキルなし・ありで書き比べる

条件を揃えるため、別セッションのサブエージェント 2 体に同じテーマを与えた。テーマは「LLM が長い会話の途中で、最初に与えた指示を守らなくなるのはなぜか。どう対策するか」。片方には何も指定せず、もう片方には SKILL.md を読んでから書くよう指示した。

先に言っておくと、スキルなしの出力も予想したほどひどくはなかった。「重要なのは」を連発する典型的な LLM 文は出ず、構成も筋が通っていた。それでも並べると差は出る。

スキルなしの出力（抜粋）。

> LLMは会話の全履歴をコンテキストウィンドウという固定長の入力として毎回読み直している。記憶を持っているわけではない。（中略）
>
> つまり「守らなくなる」のではなく、「見えにくくなる」あるいは「もう見えていない」のが実態だ。
>
> 対策は指示を一度きりの発言にしないことに尽きる。

スキルありの出力（抜粋）。

> その理由は「モデルが指示を忘れる」ではない。
> LLM は会話の全履歴を**コンテキストウィンドウ**（モデルが一度に読める入力の範囲）として毎回読み直しており、履歴が消えたわけではないからだ。
> 問題は、履歴の中での指示の比重にある。
> 会話が進むほど、コードの差分、エラーログ、直近のやり取りが履歴に積み重なり、冒頭の指示は入力全体のごく一部になる。
> （中略）
>
> もちろん、これらで逸脱が必ず防げるわけではない。

差分は三つの層に分けられる。

まず整形。一文一行になり、初出の術語が太字と丸括弧の同格挿入で定義されている。これは規範の「整形」章がそのまま反映された、機械的に検証できる差分だ。

次に論証。スキルなし版が「つまり〜が実態だ」と言い換えでまとめるのに対し、スキルあり版は読者が立てそうな誤った解釈を「モデルが指示を忘れる」と正確に書き出してから否定し、「比重の低下」という機構を一文で示す。規範の「否定する命題そのものを『』で書き出す」「因果を主張するときは機構を一文で示す」が効いている。

最後に主張の強度。スキルあり版は「必ず防げるわけではない」と条件付きで着地する。「検出・保証・解決を『必ず』できるかのように書かない」の条項どおりで、断定を増やす方向ではなく、正確さの方向に調整されているのが分かる。

書いたエージェント自身に「特に効いた条項」を報告させたところ、誤読の明示的な否定、因果の機構の一文、予告・総括の禁止の三つを挙げた。規範のどこが実際に生成へ影響するかは、この自己報告と出力の差分がおおむね一致していた。

## 使ってみての所感

効果がいちばん大きいのは、生成よりも推敲だと感じる。ゼロから書かせる場合、最近のモデルはそれなりの文章を出すので、差分は上で見たような論証の細部に現れる。一方、既存の下書きをこの規範でレビューさせると、「かもしれない」を残すべき箇所と削るべき箇所の区別、否定命題の特定、空虚な形容の指摘といった、編集者の赤字に近いフィードバックが返ってくる。

もう一つ、この SKILL.md は読み物としても価値がある。LLM に従わせる規範という体裁だが、書かれているのは人間の書き手が受けてきた編集指摘の一般形で、自分の文章の癖を名指しされる感覚がある。改変自由と明言されているので、自分のプロジェクトの文体に合わせて条項を足し引きして育てる使い方が想定されている。

なお、このスキルには読み物としての「緩急」を設計する続編 cognitive-rhythm-writing が公開されている。そちらの検証は[別記事]({% post_url 2026-07-19-cognitive-rhythm-writing-skill %})に分けた。
