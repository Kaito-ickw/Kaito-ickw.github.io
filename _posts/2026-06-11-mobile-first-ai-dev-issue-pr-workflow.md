スマホからリモートデスクトップでPCに接続して、エディタやターミナルを開き、Claude CodeやCodexに作業を指示する。動くことは動くが、操作の起点がスマホである意味がほとんどない。やっていることは「PCをリモートで操作する作業者」になることで、スマホの利点が活かせていない。

この記事では、その構成からIssue/PRベースの指示フローへの移行設計を整理する。AIエージェントをGitHub上で動かすと何が変わるか、スマホ完結に近づくためのアーキテクチャをどう考えるかを中心に書く。

---

## 今の構成の何が問題か

よくある構成はこうだ。

```
スマホ → RDP → ローカルPC → Claude Code / Codex → コード変更
```

すべての処理がローカルPCに依存している。スマホは入力デバイスとして機能しているだけで、操作の体験はPC操作そのものだ。

問題はもう一点ある。エージェントの実行がPCに縛られているため、PC電源が落ちていたり別の作業中だったりすると進まない。エージェントが「待機状態」になれる場所がない。

---

## 設計の転換点

この問題を解決するには、構成の考え方を変える必要がある。

今の構成：スマホ → PCを操作する → PCでエージェントを動かす

目指す構成：スマホ → GitHubに指示する → エージェントが動く

スマホを「開発端末」ではなく「承認・指示端末」として位置づけ直すことが設計の核心になる。

エージェントはGitHubのIssueを読み、ブランチを切り、実装してPRを作る。人間はスマホからIssueを作り、PRのdiffを確認し、コメントで修正指示を出し、mergeする。PCは「手で操作する場所」ではなく、処理を実行するワーカーに後退する。

```
スマホ → GitHub Issue/PR
                ↓
         GitHub Actions（CI・エージェント実行）
                ↓
         エージェント（Issue読む → branch作る → PR作成）
                ↓
         スマホでdiff確認 → コメントで修正 → merge
```

この構成に移行すると、スマホ側で必要なのはGitHub Mobileと、設計相談用のAIチャットアプリくらいになる。

---

## Issue/PR ベースの3層設計

構成を整理すると3つの層になる。

**指示層（スマホ）**
GitHub MobileからIssueを作り、エージェントへの指示を書く。差分の確認、コメントによる修正指示、mergeもこの層で行う。AIチャットアプリとの設計相談もここだ。

**調整層（GitHub）**
Issue、PR、GitHub Actions、Secrets、ブランチ保護などが置かれる層。エージェントが作ったコードはここを通過してCIを受ける。すべての変更がgitの履歴として残り、スマホからでも確認できる。

**実行層（エージェント）**
Issue/PRのメンションをトリガーに動作するエージェント。Claude Code GitHub Actionsはこの層の代表的な実装で、Issue/PRへの `@claude` メンションから実装・PR作成などを実行する。

この3層構成にすることで、スマホ側の操作がIssue/PRのテキスト入力だけに絞られる。

## ユースケース：チャット議論からブログ記事を生成する

Issue/PRベース構成が特に向いているのが、ChatGPT・Claude・Geminiなどとの議論をブログ記事に転換するフローだ。

### 基本の流れ

1. スマホでAIと設計や技術の壁打ちをする
2. 共有リンクまたは会話本文をGitHub Issueに貼る
3. 記事作成コマンドをコメントする
4. エージェントがドラフトをPRとして作成する
5. スマホでdiffを確認する
6. コメントで修正指示を出す
7. mergeしてGitHub Pages公開

### 共有リンクだけに頼らない

ChatGPT・Claude・Geminiの共有リンクはエージェントから安定してアクセスできない場合がある。ログイン要件、動的レンダリング、アクセス制限などで取得が失敗しやすい。

Issueテンプレートには共有リンク欄と会話本文の貼り付け欄を両方用意するのが現実的だ。リンク取得に失敗しても、本文があれば処理を続けられる。

```yaml
# Issue テンプレート例
## Source
Chat link:
Raw transcript:

## Article Goal
## Target Reader
## Tone
## Command
/blog-draft
```

### 生成先はPRのブランチに置く

生成した記事は `_posts/` に直接入れず、PRのブランチに置く。

Issueから直接 `_posts/` へ入れると、確認のステップを飛ばしてそのまま公開されるリスクがある。PRにしておけば、スマホからdiffを確認して問題箇所にコメントし修正を依頼できる。mergeして初めて公開、という流れが自然に作れる。

### チャットをそのまま記事にしない

会話ログはそのまま記事にはなりにくい。壁打ち中は推測、前提の確認、途中で変わった意見が混在している。エージェントに「この会話を記事にして」と投げるだけでは、要約をそのままMarkdownに変換したものが返ってくることが多い。

エージェントへの指示には、会話から何を記事化するかという方針と、未検証の情報の扱いを明示しておく必要がある。Issueテンプレートに「記事のゴール」「想定読者」「社内情報・個人情報の除外」といった項目を用意しておくと、指示が安定する。

---

## 人間の承認を残すべき境界

スマホ完結を目指すとき、「人間の確認ポイントまで消してしまう」のが最も危ない。

最低限、以下はエージェントに自動実行させない設計にする。

- mainへのmerge
- secretsやAPIキーの変更
- デプロイ（特に本番）
- 課金が発生する処理
- GitHub Actionsのworkflowファイル自体の変更

Issue・PR・コメントの内容はそのままエージェントへの入力になる。外部から悪意ある文字列が紛れ込む経路になりうるため、権限設計は最小限にしておくべきだ。GitHub Actions上でエージェントを動かすなら、ブランチ保護とアクセス権の設計は最初に固めておく。

---

## 最小構成から始める

いきなり完全自動化を目指す必要はない。以下の4点を整えるだけで、Issue/PRベースのフローは十分機能する。

**AGENTS.md**
エージェントへの繰り返し説明をなくすため、プロジェクトの構造・開発ルール・ワークフロー方針をここに書いておく。「mainに直接pushしない」「secretsは変更しない」「不明点は実装前に質問する」といった制約を明記すると安全性が上がる。

**Issueテンプレート**
記事生成依頼用の入力フォームを作る。Source（共有リンクまたは本文貼り付け）・記事のゴール・想定読者・トーンを毎回入力する形にすると、エージェントへの指示が安定する。スマホからでも入力しやすい項目設計が重要だ。

**PRテンプレート**
変更内容・確認方法・リスクをPR本文に書く形式にしておく。スマホでのレビューがしやすくなるだけでなく、エージェントが何を確認してほしいかを示す場所にもなる。

**CI**
PRを出した時点でlint/buildが走る状態にしておく。GitHub Pagesならjekyllのビルドを確認するだけでも効果がある。スマホでCI結果を確認してからmergeする習慣を作ると、壊れた状態のまま公開するリスクが下がる。

---

## RDPはゼロにしなくていい

リモートデスクトップを完全になくす必要はない。「ローカルで実際に動作確認する必要がある場面」や「エージェントがどうにもならない問題が起きたとき」のための例外手段として残しておく使い方がちょうどいい。

[Tailscaleを使えば自宅PCへのVPN接続が簡単に維持できる]({% post_url 2026-06-10-tailscale-guide %})。日常はIssue/PRベースで動き、例外時だけTailscale経由でRDP接続するという二段構えは、現実的な移行パスだ。

Issue/PRベース構成が整ってくると、[Androidからは Remote Control や SSH など複数のアクセスパターンがあり]({% post_url 2026-06-09-claude-code-android-patterns %})、用途に応じて使い分けられる選択肢が広がる。
