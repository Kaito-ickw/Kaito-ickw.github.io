OpenAIが公開したGPT-5.6向けのプロンプトガイドを読むと、中心にあるのは細かなテクニックではない。モデルへ渡す指示を増やすのではなく、重複を削り、達成してほしい成果と判断に必要な境界を明確にする、という考え方である。

対象は単発の質問文だけではない。システムプロンプト、ツールの説明、AIエージェントの行動規則、長時間動くワークフローまで含まれる。

この記事では、OpenAIの公式ガイドを日本語で逐語訳するのではなく、実務でプロンプトを設計・移行するときに使いやすい形へ整理する。内容は2026年7月12日に確認した公式ドキュメントに基づく。

## 結論を先に

GPT-5.6向けのプロンプトでは、次の要素を短く明示する。

| 要素 | 書く内容 |
| :--- | :--- |
| Goal | ユーザーから見て何が達成されるべきか |
| Success criteria | 何を満たせば完了と判断できるか |
| Constraints | 安全性、権限、根拠、業務上の制約 |
| Tools | どの状況で何を使うか |
| Output | 必要な形式、長さ、含める情報 |
| Stop rules | 続行、再試行、質問、中止を判断する条件 |

ただし、作業手順を一から十まで固定する必要はない。到達点と境界さえ示せば、その範囲内の探索やツール選択はモデルに任せられる。

```text
Goal:
調査結果をもとに、ユーザーが採用可否を判断できる回答を作る。

Success criteria:
- 結論を最初に示す
- 判断を支える根拠と重要な注意点を含める
- 確認できなかった点は未確認と明記する

Constraints:
- 取得した情報だけを事実として扱う
- 外部への書き込みは承認を得てから行う

Stop rules:
- 必要な根拠が揃ったら回答する
- 必須情報が不足している場合は、最小限の追加情報を尋ねる
```

これは固定の完成形ではなく、複雑なタスクを整理するための出発点である。動作を変えない説明まで埋める必要はない。

## 最初にプロンプトを短くする

既存のプロンプトをGPT-5.6へ移す場合、まず追加ではなく削除を検討する。

たとえば、同じ規則を言い換えただけの記述、結果を左右しない例、モデルがもともと安定してこなせる処理の手順書き、今回のタスクと関係ないツールの説明あたりは削って構わない。逆に成果物、成功条件、停止条件、安全性や権限まわりの制約、出力形式、検証方法はそのまま残す。

厄介なのは矛盾だ。「自律的に最後まで進める」と「操作の前に必ず確認する」が別々の場所に書かれていると、モデルはどの操作で止まるべきか判断できなくなる。GPT-5系のモデルはプロンプト上の契約を細かく守ろうとする分、説明不足よりも競合する規則の方が挙動を不安定にすることがある。

公式ガイドでは、内部のコーディングエージェント評価において、システムプロンプトを簡素化した構成が評価スコアとトークン効率の両方を改善した例も示されている。ただし、この数値は一般的な保証ではない。自分のアプリケーションを代表するタスクで評価する必要がある。

## 手順ではなく成果と停止条件を書く

複雑なタスクほど「最初にA、次にB、その後C」と手順を書きたくなる。しかし、途中の検索結果やツールの応答によって適切な経路が変わる仕事では、固定手順がかえって遠回りになる。

プロンプトでは、次のような判断基準を与える方が扱いやすい。

```text
利用できる根拠から依頼を解決する。
必要な情報が揃った時点で回答し、表現を改善するためだけの検索は増やさない。
必須の根拠が不足している場合は、不足している事実を示し、最小の代替手段を試す。
```

`ALWAYS`や`NEVER`に相当する絶対的な表現は、安全規則や禁止操作など、本当に例外がない条件へ絞る。検索するか、質問するか、もう一度試すかといった判断には、絶対命令ではなく条件を書く。

## 人格と協働スタイルを分ける

対話型のサービスでは、人格と協働スタイルを分けて考える。

人格は文章の印象を左右する部分で、温かさ、率直さ、形式性、ユーモアといった要素を指す。一方の協働スタイルは、いつ質問するか、どこまで仮定して進めるか、検証結果をどう伝えるか、不確実性をどう扱うかという、進め方そのものに関わる。

たとえば「親切に答える」だけでは、文章上の選択が曖昧である。次のように具体化する。

```text
結論から述べる。ユーザーが問題を報告した場合は、具体的な問題を短く確認してから次の手順を示す。一般的な称賛や不要な結びの挨拶は省く。
```

回答の詳しさは、APIの`text.verbosity`で全体の既定値を設定し、個別のプロンプトでは、そのタスクで残すべき内容を指定する方法が紹介されている。単に「短く」と書くより、「結論、根拠、重大な注意点、次の行動は残し、導入や重複を削る」と優先順位を示す方がよい。

## 自律性と承認の境界を一か所に置く

AIエージェントへ複数ステップの仕事を任せるなら、依頼がどこまでの操作を許可しているかを定義する。

説明や調査の依頼なら、資料を読んで結果を報告するところまでで止め、変更には踏み込まない。修正や実装の依頼であれば、対象範囲内のローカル変更と壊れない範囲の検証までは進めてよい。外部サービスへの書き込みや破壊的な操作、購入、依頼範囲を大きく超える作業になったところで、承認を求める場面になる。

この規則は一か所へまとめ、同じ「確認する」「変更しない」を何度も書かない。安全なローカル操作まで毎回止まる原因になるためである。

この考え方は、モデルの外側で権限や検証を制御する[ハーネスエンジニアリング]({% post_url 2026-06-08-harness-engineering-guide %})にもつながる。プロンプトだけで安全性を担保するのではなく、実行環境側の制約と組み合わせる必要がある。

## ツールは必要なものだけを見せる

ツールの説明には、機能、利用する状況、重要な戻り値、エラー時の挙動を書く。利用しないツールまで常に公開すると、選択肢と説明が増え、プロンプトも膨らむ。

複数の読み取りが独立していれば並列化し、前の結果によって次の行動が変わるなら順番に実行する。検索結果が空、部分的、または不自然に少ない場合は、すぐに「存在しない」と結論づけず、意味のある代替手段を一つか二つ試す。

Programmatic Tool Callingは、複数の結果をコードで絞り込み、小さな構造化結果へ変換する処理に向く。たとえば、大量レコードのフィルタリング、重複排除、集計、機械的な検証である。一回の呼び出しで済む場合、途中で意味的な判断が必要な場合、承認や引用を維持する必要がある場合は、通常のツール呼び出しの方が扱いやすい。

## 根拠の扱いをプロンプトに含める

検索を使う回答では、「引用を付ける」だけでなく、何に根拠が必要か、どの程度調べれば十分か、見つからない場合にどう振る舞うかを決める。

- 取得した情報源だけを引用する
- 主張の近くに根拠を置く
- 情報源から直接分かる事実と推測を分ける
- 情報源が食い違う場合は、その対立を示す
- 根拠が足りなければ、推測で埋めず回答範囲を狭める

検索で見つからなかったことは、その事実が存在しない証明にはならない。調査回数を無制限に増やす必要もないため、必須の所有者、日付、ID、出典などが欠けているときだけ追加検索する、といった取得予算を設ける。

## 長時間タスクでは状態を増やしすぎない

長時間のエージェント作業では、開始前に短い予告を出し、その後は主要な段階が変わったときに結果ベースで更新する。日常的なツール呼び出しを一つずつ実況させる必要はない。

コンテキストの圧縮は毎ターンではなく、大きな区切りで行う。過去の推論を保持する仕組みも、目的や前提が変わっていない場合にだけ有効である。古い推論を残し続けると、トークンや待ち時間が増えるだけでなく、現在は無効な判断へ引っ張られる可能性がある。

長期タスクのコンテキストを外部へ保持する設計については、[エージェントを長く自律動作させるためのコンテキスト設計]({% post_url 2026-06-07-agent-context-management-2026 %})でも整理している。

## 推論量を上げる前にプロンプトを確認する

`reasoning`の設定は、高ければ常によいわけではない。現在の設定を基準にし、同じ設定と一段低い設定を代表的なタスクで比較する。公式ガイドでは、速度を優先する処理には`low`、バランスの取れた出発点には`medium`を使い、`high`以上は評価で明確な改善が出た場合に選ぶ考え方が示されている。

推論量を増やす前に、成功条件、前提となる情報の取得、ツールの使い分け、検証ループがプロンプトから抜けていないかを確認する。問題が指示の欠落にあるなら、計算量を増やしても根本的な解消にはならない。

## 完了条件には検証を含める

成果物を作るタスクでは、作成したことではなく、検証を終えたことを完了条件にする。

コード変更であれば、対象に近いテストや型チェック、lint、ビルドを回し、必要なら最小限のスモークテストも足す。画面や画像が絡む変更は、実際にレンダリングして欠落やはみ出し、余白、表示状態を目で確認したい。検証そのものが実行できない状況なら、その理由と代わりに何を確認したかを書いておく。

プロンプトへ「正しく確認する」とだけ書くのではなく、何を確認すればよいかを指定する。モデルが使える検証ツールも必要である。

## GPT-5.6へ移行するときの順序

既存のアプリケーションを移行するときは、モデル変更とプロンプト変更を同時にまとめて行わない。

1. モデルを切り替え、現在の推論設定は維持する
2. プロンプトを変えず、代表的な評価を実行する
3. 古い補助指示、重複、無関係なツールを削る
4. 実測した回帰を直す最小限の指示だけを加える
5. プロンプトまたは推論設定を変えるたびに、同じ評価を再実行する

一度に全面改訂すると、挙動の変化がモデル、プロンプト、ツール、推論設定、実行環境のどれに由来するか分からなくなる。少数の実際のトレースから失敗の型を特定し、原因になりそうな指示を小さく直す方が検証しやすい。

## まとめ

GPT-5.6向けのプロンプト設計で効くのは、指示を精密に積み上げることではなく、成果や制約、使える根拠、完了条件を短く示したうえで、経路の選び方はモデルに委ねることだ。

プロンプトの良し悪しは長さでは決まらない。重複や矛盾を削って権限と停止条件を一か所に集約し、検証まで完了条件に含めておく。あとは実際の利用場面に近い評価で、変更を一つずつ確かめていけばいい。
