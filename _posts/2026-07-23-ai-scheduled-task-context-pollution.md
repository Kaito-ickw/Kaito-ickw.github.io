ChatGPTやClaudeのスケジュール機能で、毎朝ニュースをまとめさせたり、市場の動きをチェックさせたりしている人は多いはずだ。最初の数回はそれらしい分析が返ってくる。ところが1週間、1ヶ月と回し続けると、内容が妙に同じ結論の言い換えになったり、とっくに変わったはずの前提を引きずったりすることに気づく。同じ指示を出しているつもりなのに、回答の質だけがじわじわ落ちていく。

原因の多くは、モデルの性能そのものではない。定期実行の裏側で「何を毎回コンテキストとして渡しているか」の設計にある。

## なぜ履歴が分析結果を歪めるのか

チャット系AIのネイティブなスケジュール機能は、実行結果を同じチャットやタスクへ蓄積する作りになっているものが多い。各社とも内部実装の詳細までは公開していないが、次のような仕様から、少なくとも一部のサービスは完全なステートレス実行ではないと考えられる。

- タスクが特定のチャットに紐づく
- 過去の実行内容を記憶する
- 結果が同じチャットへ追記される
- チャット内で行った追加指示が次回以降にも保存される

過去の出力が次回のコンテキストに含まれると、モデルはそれを「再検証すべき仮説」ではなく「既に確定した前提」として扱いやすくなる。これは会話の続きを扱うLLMの通常の挙動としては自然だが、前提が固定されるべきでないタスクとは相性が悪い。

具体的には、次のような劣化が起こりうる。前回の仮説を必要以上に維持する。古いニュースや価格水準を引きずる。過去の回答の言い換えが増える。過去の誤りを自己引用してかえって強化する。当初のタスク指示より直近の会話を優先する。新しい事実よりも、過去のストーリーとの整合性を優先する。

市場分析、ニュース監視、障害・稼働監視、競合調査、セキュリティ情報の監視、日次・週次レポートといった、毎回ゼロベースで最新状態を評価したいタスクほど、この蓄積型の実行方式との相性が悪い。

[エージェントを長く自律動作させるためのコンテキスト設計]({% post_url 2026-06-07-agent-context-management-2026 %})でも触れたように、コンテキストは長く積めば積むほど良いわけではない。定期実行の場合、積み上がる対象が「今回のタスクに必要な情報」ではなく「過去の自分の発言」になりやすい点が、単発のロングコンテキストとは異なる問題を生む。

## サービス比較: 新規セッション実行に対応しているか

2026年7月23日時点の公式ドキュメントを確認した。**「セッションが新規かどうか」と「別の永続記憶が効くかどうか」は別軸**であることが、3サービスに共通する重要な点だ。以下では、公式に明記されている仕様と、そこから考えられる推測を区別して記載する。

### ChatGPT Tasks

ChatGPTのタスクには2種類ある。「スタンドアロン」のタスクは実行のたびに新規チャットを開始し、結果はScheduledの一覧に表示される。一方、既存チャット内で作成したタスクはそのチャットへ戻り、既存の会話コンテキストを引き継ぐ。作成時にどちらで動かすかを選ぶ仕様で、公式ドキュメントに事後切り替えの設定は見当たらない（[learn.chatgpt.com/docs/automations](https://learn.chatgpt.com/docs/automations)）。

さらに「監視タスク」という種類があり、これはWebや連携アプリの変化を定期的にチェックし、前回の実行内容を記憶した上で、終了条件を満たすと自動的に停止する（[help.openai.com](https://help.openai.com/en/articles/10291617-scheduled-tasks-in-chatgpt)）。監視という性質上、ある程度の記憶は仕様として組み込まれている。

### Gemini Scheduled Actions

Geminiのスケジュールされたアクションは、「the scheduled action chat」という単一の専用チャットに紐づき、実行結果はそこへ追記される。チャット内で指示を編集すると、その変更は次回以降の実行にも反映される（[support.google.com/gemini/answer/16316416](https://support.google.com/gemini/answer/16316416?hl=en)）。この機能を使うにはKeep Activityがオンであることが前提条件になっており、ドキュメント上、各回を独立したセッションとして実行する設定は見当たらない。3サービスの中では、履歴が最も構造的に蓄積されやすい設計といえる。

### Claude Cowork / Claude Code Scheduled Tasks

Claudeには複数のスケジュール実行の仕組みがある。

Claude Coworkのスケジュールタスクは、それぞれが独立したCoworkセッションとして実行され、過去の実行内容が自動的に蓄積されることはない。ただし、保存済みの指示（プロンプト）と、接続済みのconnectorやアカウントに保存されたファイルへはアクセスできる（[support.claude.com](https://support.claude.com/en/articles/13854387-schedule-recurring-tasks-in-claude-cowork)）。

Claude Code Desktopのスケジュールタスクも、実行のたびに新しいセッションを開始する。ただしこちらはローカルのファイルへアクセスするため、作業フォルダの状態（コミットされていない変更を含む）がそのまま次回の実行に持ち越される。CLAUDE.md、`~/.claude/settings.json`のallowルール、タスクごとに保存されたSKILL.mdの内容も影響しうる。実行ごとに独立したGit worktreeを使うトグルもあり、これを有効にすれば作業フォルダの状態は分離できる（[code.claude.com/docs/en/desktop-scheduled-tasks](https://code.claude.com/docs/en/desktop-scheduled-tasks)）。

クラウド側のClaude Code Routinesは、トリガーが発火するたびに新規セッションを開始し、リポジトリはデフォルトブランチからその都度フレッシュにクローンされる。ローカルの作業状態は引き継がれない一方、リポジトリにコミットされたCLAUDE.mdやSkills、設定済みのconnectorは毎回同じものが読み込まれる（[code.claude.com/docs/en/routines](https://code.claude.com/docs/en/routines)）。

### 比較表

| サービス | 実行単位 | 過去の実行/履歴の扱い | 新規セッションの選択肢 | 影響しうる永続コンテキスト |
| :--- | :--- | :--- | :--- | :--- |
| ChatGPT Tasks（スタンドアロン） | 新規チャット | 蓄積されない | 作成時に選択可能 | Memory機能（有効な場合） |
| ChatGPT Tasks（既存チャット内） | 既存チャットを継続 | チャット履歴が毎回文脈に含まれる | 不可 | 同上 |
| ChatGPT 監視タスク | 専用の追跡 | 前回の実行を記憶し終了条件の判定に使う | 公式情報からは不明 | Memory機能（有効な場合） |
| Gemini Scheduled Actions | 単一の専用チャット | 結果が追記され、指示編集も次回へ反映 | 見当たらない | Keep Activity（必須） |
| Claude Cowork Scheduled Tasks | 独立セッション | 蓄積されない | 仕様として独立 | connector・保存済みファイル |
| Claude Code Desktop Scheduled Tasks | 新規セッション | セッション自体は蓄積されない | 仕様として独立 | 作業フォルダの状態、CLAUDE.md、allowルール、SKILL.md（worktree未使用時） |
| Claude Code Cloud Routines | 新規セッション＋フレッシュクローン | 蓄積されない | 仕様として独立 | リポジトリにコミットされたCLAUDE.md・Skills・connector設定 |

## プロンプトだけでどこまで改善できるか

Geminiのように新規セッション実行の選択肢がないサービスや、ChatGPTの既存チャット内タスクのように継続性を優先したい場面では、プロンプトによる緩和策が現実的な選択肢になる。

- 毎回の実行を独立した分析として扱うよう指示する
- 過去の生成回答・過去の結論・過去のニュース要約を事実として継承しないよう指示する
- 今回取得した最新情報と、明示された「現在の状態」だけを根拠にゼロベースで評価するよう指示する
- 過去情報を参照する場合は、継続性の確認に必要な範囲だけに限定するよう指示する
- 過去の結論と異なる結果になっても、整合性を保つために判断を変えないよう指示する

ただし、これは新規セッションを保証するものではない。会話履歴の実データがモデルへ渡っている以上、「過去を無視しろ」という指示自体も、その履歴の中の1ターンに過ぎない。指示が新しいターンで上書きされたり、長い履歴の中で相対的な重みを失ったりする可能性は残る。プロンプトは履歴の影響を弱める緩和策であって、履歴が渡らないことを保証する仕組みではない、という位置づけで捉えておくのが安全だ。

## 理想の設計: 完全な記憶なしではなく制御された状態管理

ここまでを踏まえると、目指すべきは「完全な記憶なし」ではなく「制御された状態管理」だ。毎回のセッションに渡すべきものは、次の5つに絞られる。

1. 新しいセッション
2. 固定された分析指示
3. 現在の設定や監視対象などの事実
4. 前回から引き継ぐべき未解決論点だけをまとめた短い状態データ
5. 今回取得した最新情報

逆に、原則として次回へ渡さないものは、過去の日次レポート全文、過去のモデルの推測、解消済みの議論、古いニュース記事、途中の会話や雑談、そして過去回答の文体や構成である。モデルの感想や表現のクセまで引き継ぐ必要はない。

状態は会話履歴とは別の、YAMLやJSONなどの構造化データとして管理する。たとえば競合サービスを継続的にウォッチするタスクなら、次のような形になる。

```yaml
watch_targets:
  - name: ExampleCorp
    metric: 月間アクティブユーザー数
    baseline: 120000
    alert_condition: 前月比-10%以下で警告

standing_hypotheses:
  - 競合Aの値下げは短期キャンペーンであり、継続しない可能性
  - 市場全体の成長率は鈍化傾向にある

open_questions:
  - 競合Bの新機能は解約率に影響しているか
  - 次回レポートで為替影響を切り分けるべきか

last_updated: 2026-07-22
```

各回の処理は、この状態データと今回取得した最新情報だけを入力として分析する。処理の終了時には、レポート全文ではなく、次回に必要な状態だけを更新する。前回の結論を機械的に引き継ぐのではなく、`standing_hypotheses`や`open_questions`は毎回本当に成り立つかを再評価した上で書き換える対象として扱う。

## ネイティブ機能で足りない場合の外部実装パターン

新規セッション実行を保証できないサービスや、状態管理をより厳密に制御したい場合は、APIや外部の実行基盤から毎回新しいセッションを作る方法がある。

- APIで毎回新しいthreadまたはsessionを作る
- GitHub Actionsから定期実行する
- ローカルエージェントをcronなどで起動する
- Claude CodeやCodexを毎回新しいプロセスとして起動する
- 状態をMarkdown、YAML、JSON、SQLiteなどへ保存する

Claude Code Routinesの`/fire`エンドポイントはこの発想に近い。APIトリガーを設定すると、リクエストのたびに新しいセッションIDが発行され、保存済みのプロンプトに加えて、そのリクエスト固有のテキスト（アラート内容など）を渡せる。会話の継続ではなく、固定プロンプト＋都度の入力という構成が、サービス側の仕様として用意されている一例だ（[code.claude.com/docs/en/routines](https://code.claude.com/docs/en/routines)）。

外部実装のメリットは、会話履歴と永続状態を明確に分離できることにある。セッションは使い捨てにし、次回へ引き継ぐべき情報だけを外部のファイルやデータベースに明示的に保存する。[AIコーディングを手動プロンプトからループ設計へ変える]({% post_url 2026-07-22-loop-engineering-roadmap %})で扱ったstate fileの考え方は、定期実行における履歴汚染対策としてもそのまま応用できる。

## まとめ

定期実行の設計で見るべきは、「記憶があるかどうか」ではなく「何を状態として明示的に引き継ぎ、何を毎回捨てるか」だ。ChatGPT・Gemini・Claudeはいずれも新規セッションと永続記憶を別軸で持っており、どちらの比重が大きいかがサービスごとに異なる。ネイティブ機能の仕様を理解した上で、足りない部分を状態ファイルとプロンプト設計、必要であれば外部実装で補う。それが、毎回ゼロベースで評価したいタスクを定期実行に載せるときの現実的な落としどころになる。
